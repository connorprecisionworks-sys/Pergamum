# Pergamum â€” Architecture Decisions

Decisions made during the v2 expansion. Review these if something behaves unexpectedly.

---

## Phase 1 â€” Schema (migration 0006)

### D-001: `copies` is separate from `views`
`prompts.views` already existed and tracks page/detail views.
`prompts.copies` is a new column tracking explicit "copy prompt" actions via the `/api/prompts/copy` endpoint.
The reputation trigger (`record_prompt_copy`) awards +1 rep to the *original author*, not the copier.

### D-002: Reputation floor is 0
`GREATEST(0, reputation + delta)` is applied everywhere. Reputation can never go negative, even with many downvotes or a removed prompt.

### D-003: Remix rep awarded at publish time, not submit time
`+2` for the remixer and `+3` for the original author are awarded when the remixed prompt transitions to `status = 'published'`, not when it's first submitted. This is consistent with the `+10` publish reward and means pending/draft remixes award nothing.

### D-004: Remix bonus stacks with the publish reward
Publishing a remix gives the author `+10` (standard publish) `+ +2` (remix bonus) = `+12` total. The spec listed these as separate rules; they're both applied in `award_reputation_on_publish`.

### D-005: Badge check uses live DB counts, not cached profile aggregates
`check_badges_for_user` runs `SELECT COUNT(*)` queries rather than reading from `profiles.contribution_count` etc. This avoids race conditions when multiple triggers fire in sequence on the same row. Slightly slower but always correct.

### D-006: Daily comment cap counts *inserted* comments, not rep awards
The trigger counts how many comments the user has inserted today (`created_at >= CURRENT_DATE`). Rep is awarded for the first 10. This is a simple count â€” there's no separate "rep_comments_today" column. Farming 11+ comments per day just means comments 11+ give no rep.

### D-007: `record_prompt_copy` is an RPC function, not a DB trigger
Copy tracking originates in the application (`/api/prompts/copy`). Moving it to a trigger on a hypothetical `copy_events` table was considered but rejected â€” the API already has auth validation. The server action calls `supabase.rpc('record_prompt_copy', { p_prompt_id })` after verifying the user is authenticated.

### D-008: `collections.slug` is unique per owner, not globally
`UNIQUE (owner_id, slug)` â€” two different users can both have a collection named "favourites". Globally unique slugs would require coordination at write time and offer little user benefit.

### D-009: `featured_prompt_id` on profiles uses ON DELETE SET NULL
If an admin removes a user's featured prompt, `featured_prompt_id` silently becomes NULL rather than cascading. No notification to the user â€” this is acceptable for now.

### D-010: Trigger naming â€” `votes_award_reputation` fires alongside `votes_sync_counts`
PostgreSQL fires multiple AFTER triggers on the same table alphabetically. `votes_award_reputation` (a) fires before `votes_sync_counts` (s). Since they update different tables (`profiles` vs `prompts`) there is no dependency; order doesn't matter.

### D-011: No backfill of `lifetime_upvotes_received` / `lifetime_copies`
These aggregates start at 0 for all users. Existing vote history is not backfilled. Reputation for pre-migration activity is also not retroactively awarded. This is a clean-slate for the v2 engagement layer.

### D-012: Badge `first_remix` awarded on publish, not on fork
The badge checks `COUNT(*) FROM prompts WHERE forked_from_id IS NOT NULL AND status = 'published'`. Forking a prompt and leaving it in draft does not earn the badge.

### Security hardening (prompt #8)

Ran a full security audit across six surfaces. **Service role key**: `SUPABASE_SERVICE_ROLE_KEY` appears only in `lib/supabase/server.ts` (the `createServiceClient()` definition) and in `.env.local.example`; `createServiceClient()` is only called from `app/(app)/admin/` server-side code â€” no leakage to client bundles. **JSON-LD XSS**: `app/(app)/prompts/[slug]/page.tsx` was embedding `JSON.stringify(jsonLd)` directly into a `<script>` tag; `JSON.stringify` does not escape `</script>`, so a prompt title or description containing that string would prematurely terminate the script tag, enabling script injection. Fixed by `.replace(/<\/script>/gi, "<\\/script>")` on the serialized string (a valid JSON escape). **Open redirect**: `app/auth/callback/route.ts` passed the raw `?next=` param as `${origin}${next}`, which allows protocol-relative values like `//evil.com` to redirect off-site in some browser/redirect-chain combinations. Fixed by validating `next.startsWith("/") && !next.startsWith("//")` before use, defaulting to `/dashboard`. **Copy route auth**: `app/api/prompts/copy/route.ts` had no authentication check; anonymous POST requests would silently fail due to RLS blocking the update, but the intent is clearly auth-only. Added an early `getUser()` check that returns `{ ok: true }` for unauthenticated callers rather than a 401 (preserving the fire-and-forget UX). **Rate limiting**: Added `supabase/migrations/0008_security_hardening.sql` creating a `rate_limit_vote_log` table (RLS enabled, authenticated users can only insert their own rows) and a `purge_old_rate_limit_logs()` SECURITY DEFINER function for scheduled cleanup. The vote route (`app/api/vote/route.ts`) now queries the log for entries in the last 60 seconds; returns HTTP 429 after 10 actions per user per minute. **RLS audit**: All 13 tables (profiles, categories, prompts, votes, comments, tools, reports, badges, user_badges, collections, collection_prompts, follows, analytics_events) have RLS enabled with appropriate SELECT/INSERT/UPDATE/DELETE policies; no gaps found. Markdown rendering in `comment-section.tsx` uses `react-markdown` v7+ which disables raw HTML by default â€” `remarkGfm` adds only GFM syntax and does not enable HTML injection. Build, type-check, and lint passed cleanly on all 27 pages.

### Universal prompt moderation gate

Changed the submission flow so that every prompt from a non-admin user enters `status = 'pending'` review before publishing, regardless of that user's prior contribution count. Previously, only a user's first two prompts went to review; prompts three and beyond were published immediately. The gate was purely application-layer: a single condition in `app/(app)/submit/submit-form.tsx` (`const needsReview = !isAdmin && contributionCount < 2`) determined the `status` value written on INSERT. The fix is a one-line change: `const needsReview = !isAdmin`. The `contributionCount` prop was removed from `SubmitFormProps` and its call site in `submit/page.tsx`. **Rationale**: at launch, consistent quality and a curated first impression outweigh the friction cost of making established contributors wait for approval; the 2-prompt threshold was a reasonable trust-building heuristic, but it creates a two-tier contributor experience that becomes a support surface ("why do my prompts go live instantly but my friend's don't?"). The admin publish flow, the +10 reputation trigger on `status â†’ 'published'` (awarded by the `award_reputation_on_publish` trigger in migration 0006), and the `contribution_count` trigger are all unchanged â€” they fire at approval time exactly as before. **Reversion path**: restore `const needsReview = !isAdmin && contributionCount < 2` in `submit-form.tsx`, re-add `contributionCount: number` to `SubmitFormProps`, and pass `contributionCount={profile?.contribution_count ?? 0}` from `submit/page.tsx`. No database migration is required in either direction.
